# Multi-User Co-Work Model -- Design Brief

## The Question

How does Lucy from Finance and Max from Engineering each interact with their department's agents daily, in a human-in-the-loop style?

## Current State

Today, Orchestra has one admin user who sees everything. The department selector switches context but it's the same session, same person. This works for a solo CEO but breaks when department heads need their own space.

## The Vision

```
CEO (Alex)
  └── Sees all departments, approves cross-department decisions
  
Lucy (Head of Finance)
  └── Logs in, sees ONLY Finance department
  └── Chats with her Finance agents (Bookkeeper, Analyst, Auditor)
  └── Reviews their work, approves outputs, adds context
  └── Gets morning brief from her agents
  └── Doesn't see Engineering, Design, or other departments
  
Max (Head of Engineering)
  └── Logs in, sees ONLY Engineering department
  └── Chats with Dev, QA, Ops agents
  └── Reviews code suggestions, approves deployments
  └── Triggers routines, manages engineering tasks
  └── Doesn't see Finance or other departments
```

## Key Design Principles

1. **Each department head has their own login** -- not a shared admin account
2. **Department-scoped by default** -- Lucy sees Finance, Max sees Engineering, automatically
3. **CEO sees everything** -- the admin/CEO role retains cross-department visibility
4. **Agents remember who they're talking to** -- conversations are per-user, not per-department
5. **Human-in-the-loop is the default** -- agents suggest, humans approve

## Proposed Ideas

### Idea 1: User Roles with Department Assignment

**How it works:**
- During setup (or after), create users and assign them to departments
- Roles: `admin` (CEO, sees all), `department_head` (sees their department + company goals), `viewer` (read-only)
- When Lucy logs in, the sidebar auto-selects her department. She can't switch to Engineering.
- When Alex (admin) logs in, he gets CEO View with the department selector

**Schema change:**
```
users table: add department_id (nullable -- NULL = admin/CEO)
users table: add role ('admin' | 'department_head' | 'viewer')
```

**UX impact:**
- Login page stays the same
- After login, department heads go straight to their department dashboard -- no selector needed
- CEO gets the full sidebar with department selector
- Department heads see a simplified sidebar without the department dropdown

**Complexity:** Small. 1 table column + middleware check + UI conditional.

### Idea 2: Personal Agent Chat Threads

**How it works:**
- Each user has their own conversation thread with each agent
- Lucy's chat with "Analyst" is separate from Alex's chat with "Analyst"
- The agent sees the user's name and adjusts context accordingly
- Chat history is per-user, not global

**Schema change:**
```
chat_messages: add user_id column
```

**UX impact:**
- Chat panel shows "Your conversation with Analyst" not "Department chat"
- Agent can say "Good morning Lucy, here's your daily finance summary"
- History is personal -- Lucy won't see Alex's conversations with the same agent

**Complexity:** Small. 1 column + filter on queries.

### Idea 3: Morning Brief per Department Head

**How it works:**
- Each department head gets a personalized morning brief when they log in
- The brief is generated by their department's agents (via a routine)
- Shows: overnight activity, pending tasks needing review, agent recommendations, budget status
- Displayed as a card on the dashboard or as an inbox item

**Implementation:**
- A system routine that runs at 8am for each department
- Agent generates a brief personalized to the department head
- Brief stored in a `notifications` or `inbox` table
- Dashboard shows "Good morning Lucy -- here's what happened overnight"

**Complexity:** Medium. New inbox/notification system + routine template.

### Idea 4: Approval Queues per User

**How it works:**
- When an agent needs human approval, it creates an approval request tagged to the department head
- Lucy sees only Finance approvals in her queue
- Max sees only Engineering approvals
- CEO sees all approvals across departments

**Schema change:**
```
approval_requests: add assigned_to_user_id
```

**UX impact:**
- Sidebar shows approval count badge per user
- Department heads only see their relevant approvals
- "Approve" / "Reject" / "Ask for more info" actions

**Complexity:** Small. 1 column + filter.

### Idea 5: Agent Work Review Flow

**How it works:**
- When an agent completes a task, it goes to "Review" status
- The department head sees a queue of completed work needing their review
- They can: Approve (move to Done), Request Changes (back to In Progress with feedback), or Escalate (to CEO)
- This is the daily co-work loop: agents work, humans review

**Implementation:**
- Tasks in "review" status are the review queue
- Review UI shows: what the agent did, the output, token cost, time taken
- One-click approve, or add feedback and send back
- This already exists partially (review status in kanban) -- needs to be elevated to a first-class flow

**UX impact:**
- Dashboard shows "3 items need your review" prominently
- Review queue is the primary daily interaction point for department heads
- Clicking an item shows the agent's work with approve/reject/feedback

**Complexity:** Medium. New review queue component + enhanced task detail.

### Idea 6: Department Onboarding Invite

**How it works:**
- After setup, the admin generates invite links per department
- "Invite Lucy to manage Finance" -> generates a URL
- Lucy clicks, creates her account, is automatically assigned to Finance
- She sees only her department on first login

**Schema change:**
```
invites table: id, department_id, role, token_hash, expires_at, accepted_at
```

**UX impact:**
- Settings > Team section with "Invite Department Head" button
- Generates a link like `https://orchestra.company.com/invite/abc123`
- Lucy clicks, sets password, logs into her department

**Complexity:** Medium. New invite flow + table.

## Recommended Approach (phased)

### Phase A (build now -- enables multi-user)
1. **User roles with department assignment** (Idea 1) -- the foundation
2. **Personal chat threads** (Idea 2) -- conversations are per-user
3. **Department-scoped view on login** (part of Idea 1) -- auto-scope by role

### Phase B (build next -- enables daily co-work)
4. **Agent work review flow** (Idea 5) -- the primary daily interaction
5. **Approval queues per user** (Idea 4) -- human-in-the-loop governance

### Phase C (build after -- enables team scale)
6. **Department invite flow** (Idea 6) -- onboard department heads
7. **Morning brief per head** (Idea 3) -- personalized agent summaries

## How Lucy's Day Looks (after all phases)

```
8:00 AM -- Lucy opens Orchestra on her phone
  -> "Good morning Lucy. 2 items need your review."
  
8:05 AM -- Reviews Analyst's overnight work
  -> "Monthly close prep report" -- [Approve] [Request Changes]
  -> Approves it. Agent marks task done.
  
8:10 AM -- Reviews Bookkeeper's reconciliation
  -> Spots an issue. Adds feedback: "Check the Q2 accrual -- looks off by $12K"
  -> Agent receives feedback, re-runs with correction
  
9:00 AM -- Chats with Analyst
  -> "Can you pull the variance report for the board meeting?"
  -> Agent runs the analysis, presents results
  -> Lucy reviews, asks for one adjustment, approves
  
2:00 PM -- Gets a notification
  -> "Auditor flagged: unusual expense pattern in Marketing department"
  -> Lucy reviews, escalates to CEO for cross-department visibility
  
5:00 PM -- Dashboard shows
  -> "Today: 4 tasks completed, 1 escalated, $0.23 in agent costs"
```

## How Max's Day Looks

```
7:30 AM -- Max opens Orchestra
  -> "3 PRs reviewed by QA overnight. 1 needs your attention."
  
7:35 AM -- Reviews QA's code review
  -> QA agent flagged a security concern in the auth module
  -> Max reads the analysis, agrees, creates a fix task for Dev
  
8:00 AM -- Triggers "Daily Standup" routine
  -> Dev reports WIP status, QA reports test coverage, Ops reports infra health
  -> Max reads the combined brief in 2 minutes
  
10:00 AM -- Chats with Dev
  -> "Implement the payment gateway integration. Here's the API docs [attachment]"
  -> Dev agent starts working, Max monitors progress
  
3:00 PM -- Ops agent alerts
  -> "Disk usage at 85% on production. Recommend cleanup."
  -> Max approves the cleanup action. Ops agent executes.
```

## Questions for CEO Review

1. Should department heads be able to see OTHER departments' agents (read-only) or strictly scoped?
2. Should the CEO be notified when a department head escalates something?
3. Should agents know who they're talking to and adjust their communication style?
4. How formal should the review flow be? (Quick approve/reject vs detailed review forms)
5. Should we support multiple users per department, or strictly one head per department?
